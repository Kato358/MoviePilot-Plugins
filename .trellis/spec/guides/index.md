# 思维指南

> **目的**: 在动手之前扩展思维，发现容易遗漏的问题。

---

## 为什么需要思维指南

**大多数 bug 来自"没想到"**, 而非技术能力不足:

- 没想到层间边界 → 跨模块 bug
- 没想到代码重复 → 到处复制粘贴
- 没想到边界情况 → 运行时异常
- 没想到防风控 → API 被封禁

这些指南帮助你在编码前 **提出正确的问题**。

## 快速参考: 思维触发点

### 何时考虑跨层问题

- [ ] 功能涉及 3+ 模块（clients、handlers、主类、UI）
- [ ] 数据格式在模块间转换（如资源格式统一化）
- [ ] 多个处理器需要同一客户端的功能
- [ ] 不确定逻辑放在哪一层

→ 阅读 [跨层思维指南](./cross-layer-thinking-guide.md)

### 何时考虑代码复用

- [ ] 写了与已有代码相似的逻辑
- [ ] 同一模式在 3+ 处出现
- [ ] 多处添加相同字段/配置
- [ ] **修改任何常量或配置时** ← 先搜索！
- [ ] **创建新的工具/辅助函数** ← 先搜索！

→ 阅读 [代码复用思维指南](./code-reuse-thinking-guide.md)

## 修改前规则（关键）

> **修改任何值之前，先搜索！**

```bash
grep -r "要修改的值" plugins.v2/p115strgmsub/
```

这个习惯能防止大多数"忘了更新 X"的 bug。

## 项目特定的思维要点

### 115 网盘风控

每次修改 API 请求相关代码时，检查:
- 是否使用了 `RateLimiter`？
- 批量操作是否有间隔？
- 是否处理了限流错误码（990001, 990002, 990009）？

### 订阅站点字段兼容性

`sites` 字段在 SQLite 中可能是字符串或 JSON 列表，修改相关代码时:
- 使用 `_guess_sites_storage_format` 动态判断
- 不能假设统一格式
